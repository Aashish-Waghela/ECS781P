Threat Notification

Ragnar Locker Ransomware deployed through Virtual Machine to evade detection

Incident

Type: Incident | Issue Date : 23-05-2020 | TLP: Green | Industry: All | Region: All | Serial: G-TN-EN-01-13022

Summary

On May 21, 2020, security researchers at Sophos Labs reported on a new ransomware attack technique that deploys the Ragnar Locker ransomware on target machine through a virtual machine (VM), in order to evade detection by local antivirus software. Ragnar Locker ransomware has been active since late December 2019, and generally seen to target large organizations and corporate networks. Ragnar Locker group is known to first exfiltrate data from compromised machines and networks before deploying their ransomware to encrypt files. In past attacks, the Ragnar Locker group has been seen exploiting exposed Remote Desktop Protocol (RDP) endpoints and compromised MSP (managed service provider) tools, in order to gain access to internal networks. After gaining initial access, the actor(s) use PowerShell and Windows Group Policy Objects (GPOs) to move laterally across the compromised network.

 

In the attack spotted by Sophos Labs, threat actor(s) leveraged a GPO task to download and install a crafted MSI package from a remote server. The MSI package had the Oracle VirtualBox hypervisor and a virtual disk image file (VDI) named 'micro.vdi', which was a stripped-down Windows XP SP3 operating system that includes a Ragnar Locker executable (file size: 49 KB). The MSI package drops all files to the folder: "C:\Program Files (x86)\VirtualAppliances\". One of the dropped files is a batch script 'install.bat,' which upon execution, will first register and run the necessary VirtualBox application extensions 'VBoxC.dll', 'VBoxRT.dll', and the VirtualBox driver 'VboxDrv.sys'. The script then stops the Windows Shell Hardware Detection service, in order to disable the Windows AutoPlay notification functionality to facilitate the silent installation of the VirtualBox image. The script then configures the virtual machine to have access to all local and shared disk drives (such as removable drives and mapped network drives), and files stored on the infected machine. It then builds the 'micro.xml' VirtualBox configuration file, which is required to start the 'micro.vdi' virtual machine. Following this, the script goes on to delete the shadow volume copies from the infected machine to prevent file recovery and terminates a list of running processes. The terminated processes and services include business applications, database, remote management, and backup applications. This is so that any files that are open are unlocked and become accessible for encryption. These are tailored to the victim organization's network environment, including process and service names belonging to endpoint protection software. The script then mounts accessible network shares to available drive letters.

 

Once the virtual machine is setup, the script finally launches it (micro.vdi) by starting VirtualBox in headless mode: "C:\Program Files (x86)\VirtualAppliances\app64\VBoxHeadless.exe" –startvm micro -v off. This causes the batch file 'vrun.bat' located in the 'Startup' folder inside the virtual machine to be executed. This script loads the ransomware executable placed under "C:\vrun.exe" within the virtual machine image. Since the 'vrun.exe' ransomware executable is executing inside the virtual guest machine, its process and behaviors are unhindered by the local host security software on the physical host machine. The data and files accessible on the physical machine (and on any drives connected to the physical machine) are encrypted by the legitimate "VboxHeadless.exe" process, which is the VirtualBox virtualization software, since the ransomware process is now executing from within the context of the virtualization software. Researchers also noted that the ransomware is compiled exclusively per victim, as the ransom note it drops contains the victim's name.

 

According to researchers at Sophos Labs, this is the first time a ransomware group has abused a virtual machine in an attack. Threat actor(s) opted to hide the Ragnar Locker ransomware executable inside a virtual machine rather than running directly on the victim's machine, so that it provides a safe environment for the ransomware to operate outside the reach of antivirus software. Since the encryption process is run from a legitimate virtual machine, antivirus software failed to detect it.

Recommendations

Deloitte CTI recommends users

·        At present IOCs are not available as the files used by the threat actors for the compromise are custom-compiled for the victims.

·        Organizations should properly configure the services accessible through open internet like RDP,SSH etc. Threat actors are using these services as initial entry points. Strong password policies, remote access policies and systems should be patched with respect to the latest releases for these services.

·        Threat actors are exploiting GPO policies for the access. Proper implementation of GPO is required like Group Policy Object (GPO) based corporate email controls to block users from enabling macros in Microsoft Office attachments typically used for malware downloaders/droppers - including both Word documents and Excel spreadsheets - originating from outside your network.

·        Threat actor is installing a MSI package which contains the VirtualBox hypervisor and image file. Hence the monitoring team can look for the downloaded MSI files for the analysis.

·        It is also recommended to check on the endpoints that any file named vrun.exe is present in the C drive or not.
